# Step C 명세서: 개별 제품 리뷰 자동 생성

> **문서 성격:** 개발 명세서 (1회 작성, 지속 참조)
> **작성일:** 2026-02-09
> **선행 조건:** RUNBOOK v3.2 (Step A + Step B 완료), A2(중고가) 제거 반영

---

## 1. 목적 및 전략

### 1-1. 이 글의 역할

개별 제품 리뷰 글은 **제휴마케팅 전환 글이 아니다.** 사이트의 SEO 체력을 높이는 서포트 글이다.

```
역할 1: 롱테일 키워드 검색 유입 확보
    → "조지루시 가습기 리뷰", "케어미스트 가습기 단점" 등
    → 비교 글이 못 잡는 제품명 특정 검색어를 커버

역할 2: 내부 링크로 비교 글(Step B) 강화
    → 개별 리뷰 → 비교 글 링크
    → 비교 글 → 개별 리뷰 링크 (양방향)
    → 구글이 인정하는 Topical Authority 구축

역할 3: 사이트 콘텐츠 볼륨 확보
    → 카테고리당 비교 글 1개 → 비교 글 1개 + 개별 리뷰 3개 = 4개
    → 사이트 전체 페이지 수 × 4 증가
```

### 1-2. 이 글에 없는 것

| 없는 것 | 이유 |
|---------|------|
| 쿠팡 CTA ("최저가 확인하기") | 전환 글이 아님. CTA는 비교 글(Step B)에만 |
| 다른 제품과의 비교 | 비교는 비교 글의 역할. 여기서는 이 제품만 깊게 |
| 중고 환급/잔존가치 | A2 제거됨 |
| 제휴마케팅 공시 문구 | CTA 없으므로 불필요 |

### 1-3. 이 글의 포지셔닝

실제 사용 리뷰가 아니라 **데이터 기반 종합 분석**임을 명확히 한다.

```
❌ "조지루시 가습기 리뷰"
    → 실사용 후기를 기대하게 만듦. 본문에서 실망.

⭕ "조지루시 가습기, 사도 될까? — 리뷰 48건 분석 정리"
    → 데이터 기반임을 제목에서 명시. 오히려 차별점이 됨.
    → 블로거 1명 주관적 리뷰 vs 48건 종합 분석
```

## 2. 입력/출력 구조

### 2-1. 데이터 흐름

```
tco_{CATEGORY}.json
    │
    ├─ Step B  → /가습기/                    ← 3종 비교 글 (기존)
    │
    └─ Step C  → /가습기/조지루시-EE-RQ/      ← 개별 리뷰 글 (신규)
                 /가습기/케어미스트-CM50M/     ← 개별 리뷰 글 (신규)
                 /가습기/스텐팟-3L/           ← 개별 리뷰 글 (신규)
```

**입력:** `tco_{CATEGORY}.json`의 `products[i]` (1개 제품)
**출력:** `data/exports/review_{CATEGORY}_{PRODUCT_SLUG}.html` (제품당 1개)

### 2-2. 사용 가능 데이터 (A2 제거 후)

```
products[i] 에서 추출:

├─ 기본 정보
│   ├─ product_name             → 제품명
│   ├─ brand                    → 브랜드
│   └─ purchase_price           → 구매가
│
├─ tco
│   ├─ purchase_price           → 구매가
│   ├─ expected_repair_cost     → 1회 기대 수리비
│   ├─ expected_repair_cost_3yr → 3년 수리비
│   └─ as_turnaround_days       → AS 평균 대기일
│
├─ repair_stats
│   ├─ failure_types[]          → 고장 유형별 (type, probability, cost)
│   ├─ sample_count             → 수리 후기 수집 건수
│   └─ notes                    → AS 관련 특이사항
│
├─ maintenance
│   └─ tasks[]                  → 유지관리 항목 (name, frequency, automated)
│
└─ review_insights
    ├─ total_reviews            → 리뷰 분석 건수
    ├─ purchase_motivations[]   → 구매동기 (keyword, mention_count, verdict)
    ├─ satisfaction_keywords[]  → 만족 키워드 TOP
    ├─ dissatisfaction_keywords[] → 불만 키워드 TOP
    ├─ hidden_differentiator    → 숨은 차별점
    └─ environment_splits[]     → 환경별 적합성 (condition, positive, negative)
```

### 2-3. 비교 글(Step B)과의 데이터 사용 비중

| 데이터 | 비교 글 (Step B) | 개별 리뷰 (Step C) |
|--------|-----------------|-------------------|
| TCO 요약 | ⬤ 비교표 1행 | ⬤ 항목별 breakdown 상세 |
| 고장/수리 | ◐ 기대수리비 1줄 | ⬤ 고장 유형별 확률·비용 상세 |
| 유지관리 | ◐ 체감 비교표 1행 | ⬤ 작업별 상세 설명 |
| 리뷰 만족/불만 | ◑ highlight 1줄 | ⬤ 키워드별 상세 |
| 환경 적합성 | ◐ 체크리스트 1줄 | ⬤ 환경별 positive/negative 풀 서술 |
| AS 정보 | ◑ 대기일 숫자 | ⬤ 접수 방법, 택배 수리 여부, 주의사항 |
| 구매동기 분석 | ◑ 미신깨기(카테고리 수준) | ⬤ 이 제품 구매자의 동기 분석 |
| 제품 간 비교 | ⬤ 핵심 | ✕ 없음 |
| CTA (쿠팡) | ⬤ 있음 | ✕ 없음 |

⬤ 높음 / ◐ 중간 / ◑ 낮음 / ✕ 없음

## 3. 글 구조 (5 Sections)

```
Section 0: 한줄 결론 + 이 글의 성격 (데이터 기반 분석임을 명시)

Section 1: 이 제품, 사람들은 왜 샀을까? (구매동기 분석)
    ← purchase_motivations[]

Section 2: 실사용자들의 만족과 불만 (리뷰 인사이트)
    ├─ 2-1: 만족 포인트 ← satisfaction_keywords[]
    ├─ 2-2: 불만 포인트 ← dissatisfaction_keywords[]
    └─ 2-3: 이런 환경이면 추천 / 비추천 ← environment_splits[]

Section 3: 고장·수리·유지관리 (소유 비용)
    ├─ 3-1: 주요 고장 유형과 수리비 ← failure_types[]
    ├─ 3-2: AS 경험 ← as_turnaround_days + notes
    └─ 3-3: 일상 유지관리 ← maintenance.tasks[]

Section 4: 정리 + 내부 링크 (비교 글로 연결)
    ← 이 제품이 맞는 사람 / 안 맞는 사람 + 비교 글 링크
```

---

### Section 0: 한줄 결론 + 글의 성격

**목적:** 검색 유입자에게 즉시 답을 주고, 이 글이 무엇인지 명확히 한다.

**데이터:**
- `review_insights.total_reviews` → 분석 건수
- `environment_splits` → 가장 적합한 환경 1개

**형식:**
```
{제품명}, 결론부터 말하면:
{가장 적합한 환경/사용자 유형}이라면 좋은 선택입니다.

이 글은 직접 사용 리뷰가 아닙니다.
{제품명} 실사용자 리뷰 {N}건을 수집·분석한 데이터 정리글입니다.
```

**규칙:**
- 첫 줄에 결론. 이 제품이 맞는 사람을 바로 명시한다.
- 두 번째 문단에서 "실사용 리뷰 아님"을 솔직하게 밝힌다. 이게 오히려 신뢰를 준다.
- 분량: 4~5줄.

---

### Section 1: 사람들은 왜 이 제품을 샀을까?

**목적:** 구매 검토 중인 독자가 "다른 사람들은 뭘 보고 샀지?" 궁금증을 해결.

**데이터:**
- `purchase_motivations[]` → keyword, mention_count, verdict(overrated/justified/mixed)

**형식:**
```
## {제품명}, 사람들은 왜 샀을까?

리뷰 {N}건에서 구매 이유로 가장 많이 언급된 것:

1. {keyword_1} ({mention_count}건) — {verdict 해설}
2. {keyword_2} ({mention_count}건) — {verdict 해설}
3. {keyword_3} ({mention_count}건) — {verdict 해설}
```

**verdict 해설 규칙:**
- `justified`: "실사용에서도 만족도가 높았습니다."
- `overrated`: "하지만 실사용 만족 리뷰에서는 거의 언급되지 않았어요. 기대만큼은 아닐 수 있습니다."
- `mixed`: "만족한 사람과 실망한 사람이 갈렸습니다. 환경에 따라 다를 수 있어요."

**규칙:**
- 구매동기 TOP 3~5개를 언급 순으로 나열한다.
- overrated인 항목은 비교 글의 미신깨기(Section 1-1)와 연결되지만, **여기서는 이 제품 한정**으로 서술한다. 카테고리 수준 일반화는 하지 않는다.
- 분량: 10~15줄.

---

### Section 2: 실사용자들의 만족과 불만

**목적:** "실제로 쓰면 어때?"에 대한 데이터 기반 답변.

#### 2-1. 만족 포인트

**데이터:** `satisfaction_keywords[]`

```
## 👍 만족한 점

실사용자들이 가장 많이 만족한 부분:

- **{keyword_1}**: {1~2줄 설명}
- **{keyword_2}**: {1~2줄 설명}
- **{keyword_3}**: {1~2줄 설명}
```

#### 2-2. 불만 포인트

**데이터:** `dissatisfaction_keywords[]`

```
## 👎 불만족한 점

반대로 불만이 많았던 부분:

- **{keyword_1}**: {1~2줄 설명}
- **{keyword_2}**: {1~2줄 설명}
```

#### 2-3. 환경별 적합성

**데이터:** `environment_splits[]`

```
## 🏠 이런 환경이면 추천 / 비추천

{제품명}은 사용 환경에 따라 만족도가 크게 갈렸습니다.

**✅ 이런 분께 추천:**
- {condition}: {positive 요약}
- {condition}: {positive 요약}

**⚠️ 이런 분은 주의:**
- {condition}: {negative 요약}
- {condition}: {negative 요약}
```

**규칙:**
- environment_splits의 positive/negative 패턴을 풀어 서술한다.
- 비교 글에서는 체크리스트 1줄이었던 것을, 여기서 **왜 그런지 맥락까지** 설명한다.
- 예: 비교 글 "기어다니는 아기 → 케어미스트" 1줄 → 개별 리뷰에서 "초음파 방식이라 토출구 온도가 상온 수준이어서 화상 위험이 없습니다. 다만 물입자가 미세하게 바닥에 맺힐 수 있어 미끄럼 주의." 3~4줄.
- 분량: Section 2 전체 20~25줄.

---

### Section 3: 고장·수리·유지관리

**목적:** "오래 쓰면 뭐가 문제지?"에 대한 답. 이 섹션이 다른 리뷰 블로그와의 핵심 차별점.

#### 3-1. 주요 고장 유형과 수리비

**데이터:** `repair_stats.failure_types[]`

```
## 🔧 주요 고장과 수리비

{제품명} 수리 후기 {sample_count}건을 분석한 결과:

| 고장 유형 | 발생 확률 | 예상 수리비 |
|----------|----------|-----------|
| {type_1} | {probability}% | {cost}원 |
| {type_2} | {probability}% | {cost}원 |
| {type_3} | {probability}% | {cost}원 |

3년 기준 기대 수리비: **{expected_repair_cost_3yr}원**

{가장 빈번한 고장에 대한 1~2줄 보충 설명}
```

**규칙:**
- `failure_types`를 확률 순 또는 비용 순으로 나열한다.
- "3년 기준 기대 수리비"는 `expected_repair_cost × 3`으로 산출된 TCO 수치를 그대로 사용한다.
- 확률이 낮지만 비용이 높은 항목을 별도 언급한다. ("발생 확률은 낮지만, 만약 발생하면 ~원으로 부담이 큽니다.")

#### 3-2. AS 경험

**데이터:** `tco.as_turnaround_days` + `repair_stats.notes`

```
## 📞 AS 이용 시 참고

- AS 평균 대기: **{as_turnaround_days}일**
- {notes에서 추출한 AS 관련 정보}
```

**notes에서 추출할 수 있는 정보 예시:**
- 택배 수리 가능 여부
- 서비스센터 접근성
- 보증기간 관련 사항
- 부품 수급 특이사항

notes가 빈약할 경우 AS 대기일만 기재하고 넘어간다. 없는 정보를 만들지 않는다.

#### 3-3. 일상 유지관리

**데이터:** `maintenance.tasks[]`

```
## 🧹 일상 유지관리

{제품명}을 쓰면서 해야 할 일:

| 작업 | 빈도 | 직접 해야 하나? |
|------|------|---------------|
| {task_name_1} | {frequency} | {automated? "자동" : "직접"} |
| {task_name_2} | {frequency} | {automated? "자동" : "직접"} |
| {task_name_3} | {frequency} | {automated? "자동" : "직접"} |

직접 해야 하는 작업: {수동 작업 수}개
자동화된 작업: {자동 작업 수}개

{가장 번거로운 작업에 대한 1~2줄 팁 또는 주의사항}
```

**규칙:**
- 비교 글에서는 "매일/주1회/안해도됨"으로 압축했던 것을, 여기서는 **개별 작업명과 빈도를 그대로** 보여준다.
- 가능하면 "이 작업을 안 하면 어떻게 되는지" 보충 설명을 추가한다.
  예: "물통 청소를 2주 이상 안 하면 물때가 끼어 세균 번식 원인이 됩니다."
- 분량: Section 3 전체 20~25줄.

---

### Section 4: 정리 + 내부 링크

**목적:** 글을 마무리하고, 비교 글로 자연스럽게 연결한다.

**데이터:**
- `environment_splits` → 맞는 사람 / 안 맞는 사람 요약
- `tco` → 핵심 숫자 요약
- 비교 글 URL

**형식:**
```
## 정리

{제품명}은 {맞는 환경/사용자}에게 좋은 선택입니다.
{안 맞는 환경/사용자}이라면 다른 방식을 고려해보세요.

구매가 {purchase_price}원, 3년 기준 수리비 {expected_repair_cost_3yr}원,
3년 실사용 비용은 약 **{purchase_price + expected_repair_cost_3yr}원**입니다.

---

📊 다른 제품과 3년 실비용 비교가 궁금하시면:
→ [{카테고리} 3종 TCO 비교 분석]({비교글_url})

{같은 카테고리 다른 제품 리뷰 링크}
→ [{다른제품1} 실사용자 분석]({리뷰글1_url})
→ [{다른제품2} 실사용자 분석]({리뷰글2_url})
```

**규칙:**
- 맞는 사람 / 안 맞는 사람을 1줄씩만.
- 3년 실비용 = `purchase_price + expected_repair_cost_3yr` (A2 제거로 resale 없음).
- **내부 링크 3개:**
  1. 비교 글 (필수)
  2. 같은 카테고리 다른 제품 리뷰 1 (있으면)
  3. 같은 카테고리 다른 제품 리뷰 2 (있으면)
- CTA 없음. "최저가 확인하기" 없음.
- 분량: 8~10줄.

## 4. 내부 링크 구조

### 4-1. 양방향 링크 설계

```
/가습기/ (비교 글, Step B)
    ├─ Section 3-1: "조지루시 상세 분석 →" (개별 리뷰로)
    ├─ Section 3-2: "케어미스트 상세 분석 →" (개별 리뷰로)
    └─ Section 3-3: "스텐팟 상세 분석 →" (개별 리뷰로)

/가습기/조지루시-EE-RQ/ (개별 리뷰, Step C)
    └─ Section 4: "3종 비교 →" (비교 글로)
                  "케어미스트 분석 →" (다른 리뷰로)
                  "스텐팟 분석 →" (다른 리뷰로)
```

### 4-2. Step B 비교 글 수정 사항

개별 리뷰 글이 생성되면, 비교 글(Step B)의 Section 3 제품별 상세 부분에 내부 링크를 추가해야 한다.

```
### 1. 조지루시 EE-RQ

👍 **추천**: ...
👎 **주의**: ...

| 구매가 | 수리비(3년) | 3년 실비용 |
|--------|------------|-----------|
| ...    | ...        | ...       |

📌 [조지루시 상세 분석 보기 →]({리뷰글_url})
```

이 링크는 CTA가 아니다. 쿠팡이 아닌 **같은 사이트 내부 페이지**로의 이동이다.

### 4-3. URL 구조

```
메인 도메인: https://{site}/

비교 글:    https://{site}/{category}/
개별 리뷰:  https://{site}/{category}/{product_slug}/
```

`product_slug`는 TCO JSON의 제품명에서 자동 생성한다.
예: "조지루시 EE-RQ" → `조지루시-ee-rq`

> slug 규칙은 발행 시 사이트 라우팅에 맞춰 치환한다. HTML 생성 시점에서는 placeholder를 사용한다.

## 5. SEO 최적화

### 5-1. 타겟 키워드

비교 글과 개별 리뷰 글은 **다른 검색 의도**를 타겟한다.

| 검색어 유형 | 예시 | 담당 글 |
|------------|------|--------|
| 카테고리 추천 | "가습기 추천 2026" | 비교 글 (Step B) |
| 카테고리 비교 | "가습기 비교", "가습기 TCO" | 비교 글 (Step B) |
| 제품명 리뷰 | "조지루시 가습기 리뷰" | 개별 리뷰 (Step C) |
| 제품명 단점 | "조지루시 가습기 단점" | 개별 리뷰 (Step C) |
| 제품명 고장 | "조지루시 가습기 고장" | 개별 리뷰 (Step C) |
| 제품명 전기세 | "조지루시 가습기 전기세" | 개별 리뷰 (Step C) |
| 제품 vs 제품 | "조지루시 vs 케어미스트" | 비교 글 (Step B) |

### 5-2. 제목 (title) 규칙

같은 카테고리 3개 제품이 SERP에 나란히 노출될 수 있으므로, **3개 변형 패턴을 로테이션**한다.

```
패턴 A: {제품명}, 사도 될까? — 리뷰 {N}건 분석 [{년도}]
패턴 B: {제품명} 장단점 총정리 — 실사용자 {N}건 분석 [{년도}]
패턴 C: {제품명} 3년 쓰면 얼마? — 수리비·유지비 분석 [{년도}]

예시 (가습기 카테고리):
"조지루시 EE-RQ, 사도 될까? — 리뷰 48건 분석 [2026]"         ← 패턴 A
"케어미스트 CM50M 장단점 총정리 — 실사용자 35건 분석 [2026]"  ← 패턴 B
"스텐팟 3L 3년 쓰면 얼마? — 수리비·유지비 분석 [2026]"       ← 패턴 C
```

- 제품명을 제목 맨 앞에 배치 (검색 매칭)
- "리뷰 N건 분석"으로 데이터 기반임을 명시 (CTR 차별화)
- 연도 포함 (최신성 시그널)
- 제품 순서(products[0], [1], [2])에 따라 A → B → C 패턴 순서 적용

### 5-3. meta description

```
{제품명} 실사용자 리뷰 {N}건을 분석했습니다.
구매 이유, 만족/불만 포인트, 주요 고장 유형과 수리비,
일상 유지관리까지 데이터로 정리. 3년 실사용 비용 {금액}원.
```

- 150자 이내
- 핵심 키워드 포함: 제품명, 리뷰, 고장, 수리비, 유지관리

### 5-4. 구조화된 데이터 (Schema.org)

개별 리뷰 글에 `Review` 또는 `Article` 스키마를 추가하면 검색 결과에 리치 스니펫이 표시될 수 있다.

```json
{
  "@context": "https://schema.org",
  "@type": "Article",
  "headline": "{제품명}, 사도 될까? — 리뷰 {N}건 분석",
  "description": "{meta description}",
  "author": { "@type": "Organization", "name": "{사이트명}" },
  "datePublished": "{발행일}",
  "dateModified": "{수정일}"
}
```

주의: `@type: "Review"`를 사용하면 실사용 리뷰로 오해될 수 있다. 실제 사용하지 않았으므로 `"Article"`이 더 정확하다.

## 6. HTML 출력 형식

### 6-1. 생성 파일

```
data/exports/review_{CATEGORY}_{PRODUCT_SLUG}.html

예시:
data/exports/review_가습기_조지루시-ee-rq.html
data/exports/review_가습기_케어미스트-cm50m.html
data/exports/review_가습기_스텐팟-3l.html
```

### 6-2. HTML 요구사항

Step B(비교 글)와 동일한 기본 요구사항:

- `<!DOCTYPE html>` + `<meta charset="UTF-8">` + 반응형 viewport
- 내장 CSS (외부 의존성 없음), 인라인 스타일 (네이버/티스토리 호환)
- 표: `<table>` + `overflow-x: auto`
- 섹션별 `<section>` 태그

Step C 추가 요구사항:

- `<head>`에 meta description + Schema.org JSON-LD 포함
- 내부 링크는 `<a>` 태그, href는 **placeholder** (`{비교글_url}`, `{리뷰글_url}`)로 생성.
  실제 URL은 발행 시 치환한다.
- CTA 버튼 스타일 **없음**. 내부 링크는 일반 텍스트 링크 스타일.

### 6-3. 디자인 톤

비교 글(Step B)보다 **가볍고 읽기 쉬운 톤**.

- 비교 글: 표와 비교 데이터 중심, 의사결정 지원
- 개별 리뷰: 서술형 중심, "이 제품에 대해 알아야 할 것" 가이드

### 전제 조건

```
- data/exports/tco_{CATEGORY}.json 존재 (Step A4 완료)
- data/exports/blog_{CATEGORY}.html 존재 (Step B 완료)
```

### 실행

```bash
# 카테고리의 모든 제품에 대해 개별 리뷰 생성
# Claude Code가 현재 세션에서 직접 수행

1. tco_{CATEGORY}.json 읽기

   a. 해당 제품 데이터 추출

```

> **제품 단위 독립 생성 원칙:** 각 제품의 HTML은 독립적으로 생성한다. 1개 제품 생성에 실패해도 나머지 제품은 정상 출력한다.

### Step C 체크리스트

```
[ ] 제품당 review_{CATEGORY}_{PRODUCT_SLUG}.html 생성 확인 (3개)
[ ] 각 리뷰 글:
    [ ] 5개 Section 구조 확인
    [ ] 모든 숫자가 TCO JSON과 일치
    [ ] CTA(쿠팡 링크) 없음 확인
    [ ] 내부 링크 placeholder 존재 (비교 글 + 다른 리뷰)
    [ ] meta description + Schema.org JSON-LD 포함
    [ ] "실사용 리뷰가 아님" 명시 확인 (Section 0)
    [ ] 제품 간 비교 내용 없음 확인 (단독 분석만)
[ ] 표현 안전 가이드라인:
    [ ] 제품/브랜드 폄훼 표현 없음
    [ ] 데이터 출처 명시 ("리뷰 N건 분석")
    [ ] 단정 표현 없음
[ ] (선택) 비교 글에 개별 리뷰 내부 링크 추가 확인
```

## 9. 비교 글(Step B)과의 연동

### 재생성 방식 (HTML 파싱 없음)

기존 비교 글 HTML을 파싱·수정하지 않는다. 대신 **Step B를 재생성할 때 Step C 파일 존재 여부를 체크**하여 내부 링크를 포함시킨다.

```
첫 실행:  Step B (링크 없이) → Step C (3개 리뷰 생성)
재생성:   Step B (리뷰 링크 포함) ← Step C 파일 존재 확인
```

Step B 재생성 시, Section 3 제품별 상세에 아래 내부 링크를 포함한다:

```html
<h3>1. 조지루시 EE-RQ</h3>
<p>👍 추천: ...</p>
<p>👎 주의: ...</p>
<table>...</table>
<a class="cta-button" href="{cta_link}">최저가 확인하기</a>
<p>📌 <a href="{리뷰글_url}">조지루시 EE-RQ 상세 분석 (리뷰 48건) →</a></p>
```

이 링크는 CTA가 아니다. 쿠팡이 아닌 **같은 사이트 내부 페이지**로의 이동이다.

> RUNBOOK Step B 명세에 조건 추가: "Step C 파일(`review_{CATEGORY}_*.html`)이 존재하면 Section 3에 내부 링크를 포함한다."

## 10. 향후 확장

### 10-1. A5 제품별 보충 검색 (v2)

v1에서 Step C가 사용하는 데이터는 기존 A5(카테고리 수준 리뷰 수집)에서 나온다. 콘텐츠 depth가 부족할 경우, A5에 **"A5-2: 제품별 심층 검색"** 서브스텝을 추가한다.

```
A5-2 추가 검색어 (제품당 2~3회):
- "{제품명} 1년 사용기"
- "{제품명} 고장 수리 후기"
- "{제품명} 장기 사용 후기"
```

이렇게 하면 Step C는 여전히 JSON만 읽으면 되고, 데이터 수집(A5)과 콘텐츠 생성(C)이 분리된 상태를 유지한다. v1 결과물을 확인한 후 depth 부족이 확인되면 도입한다.

### 10-2. 제품 간 1:1 비교 글 (미래)

개별 리뷰가 쌓이면, "조지루시 vs 케어미스트" 같은 **1:1 비교 글**도 자동 생성할 수 있다. 이때도 같은 TCO JSON에서 2개 제품을 뽑아 생성한다.

```
/가습기/조지루시-vs-케어미스트/
```

이 글은 "조지루시 vs 케어미스트" 같은 비교 검색어를 타겟한다.

### 10-3. 카테고리 가이드 글 (미래)

"가습기 고르는 법", "가습기 종류별 차이" 같은 정보성 가이드 글도 TCO JSON의 category_insights에서 생성할 수 있다. 카테고리 수준의 일반 정보를 다루며, 비교 글과 개별 리뷰 글 모두에 내부 링크를 건다.

```
사이트 구조 (완성 시):

/{category}/                       ← 3종 TCO 비교 (Step B)
/{category}/{product_slug}/         ← 개별 리뷰 (Step C)
/{category}/{product1}-vs-{product2}/ ← 1:1 비교 (향후)
/{category}/guide/                  ← 카테고리 가이드 (향후)
```
